🔒 Analyst Response Guide — Token Theft
Phase 1 — Immediate  ⏱ 0–15 min
Step 1 — Understand what was detected
Why: The two detection types indicate different attack vectors with different root causes. Knowing which one fired determines how you investigate and what you block.
  • anomalousToken: A token was presented from a device or IP inconsistent with where it was issued. The attacker captured a browser session cookie after MFA was completed (pass-the-cookie attack) and is replaying it from a different machine.
  • tokenIssuerAnomaly: The token metadata does not match expected issuer claims. This is the primary signal of an AiTM (Adversary-in-the-Middle) phishing proxy such as Evilginx or Modlishka — the proxy sat between the user and the login page, intercepted the token post-MFA, and passed a re-signed version to the attacker.
  • Check the Detection Type column above to confirm which type fired and plan your response accordingly.
🔑 Also determine: What type of token was stolen?
Why: Token type determines remediation depth. A stolen PRT survives a password reset and requires device-level action. A stolen access token expires in ~1 hour on its own but must still be cut immediately.
  • Access token: Short-lived (~1 hr). Expires on its own, but revoke the session immediately to end the active attacker session now rather than waiting.
  • Refresh token: Long-lived with no automatic expiry. This is what Revoke sessions in Entra cancels. Without revocation the attacker silently obtains fresh access tokens indefinitely in the background.
  • Primary Refresh Token (PRT): Device-bound and the most persistent. Survives password reset entirely. Requires disabling or wiping the device in Entra to invalidate. Key signal: the attacker session shows the same deviceId as the legitimate user but originates from a different IP address.
Step 2 — Revoke the stolen token immediately
Why: This is the single most urgent action. The attacker has a valid token and an active session right now. Every minute of delay is more data they can access.
  • Click User Profile for the affected user above → click Revoke sessions. This invalidates all refresh tokens within 1 minute.
  • To also invalidate the Primary Refresh Token (PRT): in Entra → Devices, mark the affected device as non-compliant or disable it.
  • After revocation, the attacker session will fail on the next token refresh attempt (typically within 1 hour for access tokens).
Phase 2 — Blast Radius Assessment  ⏱ 15–60 min
Step 3 — Identify what the attacker accessed during the stolen session
Why: With a valid token, the attacker had access to every app and resource the user can access — email, Teams, SharePoint, OneDrive, and any other Microsoft 365 service. Knowing what was accessed determines whether a data breach notification is required.
  • Click Sign-in Logs for the affected user → look at the time window between the detection timestamp and when you revoked the session.
  • Note every application accessed and every IP the token was used from — the attacker IP will differ from the user's normal IP.
  • Check Exchange/Teams/SharePoint audit logs for file downloads, email reads, message access, or any API calls made during the attacker session.
Step 4 — Force clean re-authentication for the real user
Why: After revocation the legitimate user needs to re-authenticate. Ensure they do so from a known-good, compliant device — not the potentially compromised one.
  • Apply a Conditional Access policy requiring a compliant or Hybrid Azure AD-joined device for re-authentication.
  • Set Conditional Access sign-in frequency to 1 hour for the affected user during the investigation period.
  • Require phishing-resistant MFA (FIDO2 or Windows Hello) for re-authentication — standard OTP/SMS is bypassable by the same AiTM proxy that was used.
Phase 3 — Persistence Check  ⏱ 1–4 hrs
Step 5 — Hunt for attacker persistence mechanisms
Why: A sophisticated attacker will use the stolen token to create persistent access before it is revoked. Even after revocation, they may still be inside via a different path.
  • New MFA methods: Entra → Users → [user] → Authentication methods — any new phone number, authenticator app, or FIDO key registered after the detection time is an attacker backdoor.
  • OAuth app consents: Entra → Enterprise Applications → filter by user — attacker may have consented to a malicious app with Mail.Read or Files.ReadWrite permissions.
  • Inbox forwarding rules: Exchange Admin → Mailboxes → [user] → Mailbox Features — look for rules that silently forward all mail to an external address.
  • New registered devices: Entra → Devices — a device registered during the attacker session can obtain its own PRT, surviving a password reset.
Step 6 — If tokenIssuerAnomaly: investigate the phishing campaign
Why: An AiTM attack requires the user to visit a phishing proxy. Other users in your tenant may have received the same phishing email and been compromised without triggering a detection.
  • Check the affected user's inbox for the phishing link they clicked — usually a spoofed Microsoft login or document share link received shortly before the detection.
  • Extract the phishing URL and submit to Defender for Office 365 for tenant-wide blocking.
  • Search email audit logs for anyone else who received the same email to identify other potentially compromised accounts.
Step 7 — Harden token security to prevent recurrence
Why: Standard MFA does not protect against AiTM or pass-the-cookie attacks. Token protection and phishing-resistant MFA do.
  • Enable Token Protection: Conditional Access → Session → Token protection (preview) — binds tokens to the specific device they were issued to, making pass-the-cookie attacks fail.
  • Enforce phishing-resistant MFA: Require FIDO2 keys or Certificate-Based Auth via CA policy. These authentication methods cannot be proxied by AiTM tools.
  • Reduce token lifetimes: Conditional Access → Session controls → Sign-in frequency: 1 hour — limits how long a stolen access token remains valid.
